# GPT-5.6 Sol로 한국 변호사시험 공법 선택형 풀기

이 저장소는 `gpt-5.6-sol`이 법령·판례·교재 검색 도구를 사용했을 때 한국 변호사시험 공법 선택형 40문항을 얼마나 정확하고 안정적으로 풀 수 있는지 실험한 기록입니다.

단순히 한 번 문제를 풀어 점수만 확인한 것이 아닙니다. 실험 방법을 미리 정하고, 정답표를 숨긴 상태에서 답안을 세 차례 만들고, 모든 답안을 봉인한 뒤 마지막에 정답표를 한 번만 열어 채점했습니다.

## JuriSupport 법률 리서치 도구

이 실험의 핵심 검색 도구는 [JuriSupport 플러그인 패키지](https://github.com/jurisupport/jurisupport-plugins)에서 제공하는 한국 법률 리서치 구성요소를 활용했습니다.

- **`korean-law` MCP**: 법제처 기반 법령·판례 원문 확인
- **`legal-books`**: 사용자가 보유한 법률서적을 OCR·임베딩해 교재 근거 검색
- **`beopgoeul-search`**: 대법원 도서관 법고을 공식 사이트의 판례 검색

1차 풀이에서는 `korean-law`와 `legal-books`만 사용했고, 문항별 딥다이브에서만 `beopgoeul-search`와 시험 시점으로 차단한 판례DB를 추가로 개방했습니다.

플러그인 저장소에는 도구 코드와 설치 절차가 공개되어 있습니다. 사용자가 보유한 교재 파일과 이 실험의 교재DB 내용은 포함되지 않습니다.

> 도구 저장소: https://github.com/jurisupport/jurisupport-plugins

## 한눈에 보는 결과

| 조건 | 1회 | 2회 | 3회 | 평균 |
|---|---:|---:|---:|---:|
| 도구 없이 모델만 사용 | 33/40 | 34/40 | 32/40 | 33/40, 82.5% |
| 법령·판례·교재 도구를 사용한 전체 워크플로 | 40/40 | 40/40 | 40/40 | 40/40, 100% |

도구 없이 풀었을 때는 평균 33문제를 맞혔습니다. 도구와 다단계 검증 절차를 사용했을 때는 세 차례 모두 40문제를 맞혔습니다.

평균적으로 7문제가 개선됐으며, 정답률 차이는 17.5%포인트입니다.

## 무엇을 비교했나?

### 1. 도구 없는 조건

모델이 법령·판례·교재를 검색하지 않고 내부 지식만으로 문제를 풀었습니다.

이 조건은 세 번 실행했고 결과는 다음과 같습니다.

- 1회: 33/40
- 2회: 34/40
- 3회: 32/40
- 평균: 33/40

도구 없는 조건에서 세 번 모두 틀린 문제는 7번, 23번, 26번, 30번이었습니다.

### 2. 도구를 사용한 조건

같은 모델과 같은 추론 강도를 사용하되, 다음과 같은 검증 절차를 추가했습니다.

1. 법령·판례·교재를 검색하며 문제를 풉니다.
2. 같은 문제를 서로 다른 세 가지 관점으로 풉니다.
3. 세 답안의 다수결로 1차 답을 정합니다.
4. 검토가 필요한 문항은 법령과 판례 원문을 다시 찾아 딥다이브합니다.
5. 근거가 충분할 때만 기존 답을 변경합니다.
6. 최종 답안을 봉인합니다.

이 전체 과정을 서로 별도로 세 번 실행한 결과가 F1, F2, F3입니다.

## 왜 같은 문제를 세 가지 관점으로 풀었나?

한 번의 풀이가 특정 법리나 표현에 치우칠 가능성을 줄이기 위해 세 가지 관점을 사용했습니다.

- V1: 조문을 중심으로 순서대로 검토
- V2: 판례와 예외를 중심으로 역검토
- V3: 처음부터 중립적으로 다시 도출

문항마다 세 답안 중 두 개 이상이 선택한 답을 1차 다수결 답으로 정했습니다.

하지만 다수결만으로는 만점이 나오지 않았습니다.

| 회차 | V1 | V2 | V3 | 1차 다수결 | 최종 |
|---|---:|---:|---:|---:|---:|
| F1 | 36 | 36 | 35 | 37 | 40 |
| F2 | 36 | 38 | 35 | 37 | 40 |
| F3 | 37 | 36 | 34 | 37 | 40 |

세 회차 모두 1차 다수결 단계에서는 37/40이었습니다. 이후 딥다이브를 통해 매회 남은 오답 3개가 교정되어 최종 40/40이 됐습니다.

## 딥다이브란?

딥다이브는 시험을 다 푼 뒤 법전과 판례집을 펼쳐 한 문제씩 다시 검산하는 과정과 비슷합니다.

문항마다 새로운 독립 호출을 사용해 다음 사항을 다시 확인했습니다.

- 관련 법령과 판례 원문
- 각 선택지가 맞거나 틀리는 이유
- 1차 다수결 답에 반대되는 근거
- 인용된 근거가 실제 결론을 뒷받침하는지
- 답을 변경할 정도로 충분한 근거가 있는지

공식 정답표는 딥다이브에서도 사용하지 않았습니다. 정답표는 모든 답안을 봉인한 뒤 마지막 채점 단계에서만 열었습니다.

이번 실험에서는 `40문항 × 3회 = 120건` 중 113건을 딥다이브로 다시 검토했습니다.

- 기존 답 유지: 104건
- 기존 답 변경: 9건
- 오답에서 정답으로 변경: 9건
- 정답에서 오답으로 변경: 0건

딥다이브로 변경된 문항은 다음과 같습니다.

| 회차 | 변경된 문항 | 결과 |
|---|---|---|
| F1 | 19, 23, 26 | 모두 오답에서 정답으로 교정 |
| F2 | 7, 19, 26 | 모두 오답에서 정답으로 교정 |
| F3 | 19, 26, 30 | 모두 오답에서 정답으로 교정 |

특히 9건 중 8건은 V1·V2·V3가 모두 같은 오답을 선택한 경우였습니다.

이는 같은 모델을 여러 번 호출해 다수결만 하더라도, 모델이 같은 법리를 공통으로 잘못 이해하면 오류를 발견하지 못할 수 있다는 뜻입니다. 이번 실험에서는 법령과 판례 원문을 다시 검토하는 과정이 이러한 공통 오류를 교정했습니다.

## 결과를 어떻게 해석해야 하나?

이번 실험에서 확인된 것은 다음과 같습니다.

- 도구 없이 풀었을 때도 평균 82.5%의 기본 성능을 보였습니다.
- 단순한 세 답안 다수결은 매회 37/40이었습니다.
- 원문 중심의 딥다이브가 매회 남은 오답 3개를 교정했습니다.
- 전체 워크플로는 세 차례 모두 40/40을 기록했습니다.
- 사전에 정한 성공 기준인 세 차례 모두 만점을 충족했습니다.

따라서 이번 결과는 다음과 같이 해석하는 것이 적절합니다.

> 높은 기본 성능을 가진 모델에 법령·판례·교재 검색, 서로 다른 관점의 반복 풀이, 다수결, 원문 중심 재검토를 결합하면 이 시험에서 안정적인 만점을 재현할 수 있었다.

## 이 결과가 의미하지 않는 것

이번 결과만으로 다음 사항까지 입증된 것은 아닙니다.

- 모델이 도구 없이도 만점을 받을 수 있다는 의미는 아닙니다.
- 단순히 같은 질문을 세 번 하면 만점이 된다는 의미도 아닙니다.
- 특정 검색 도구 하나가 성능 향상 전체를 만들었다고 볼 수 없습니다.
- 다른 연도나 민사법·형사법에서도 만점이 나온다고 단정할 수 없습니다.
- 실제 시험 시간과 비용 제한에서도 같은 성능이 나온다는 의미는 아닙니다.
- 모델이 학습 과정에서 시험문제를 전혀 본 적이 없었다는 증명은 아닙니다.

이번 실험은 모델, 검색 도구, 세 가지 풀이 관점, 다수결, 딥다이브를 한꺼번에 결합한 전체 시스템을 평가했습니다. 각 구성요소의 효과를 따로 측정한 실험은 아닙니다.

## 정확도는 높았지만 효율은 낮았다

딥다이브는 전체 120건 중 113건에 적용됐습니다. 검토 대상 비율은 94.2%입니다.

그중 실제 답이 변경된 것은 9건으로, 딥다이브 1건당 답 변경 비율은 약 8%였습니다.

따라서 이번 실험은 소수의 어려운 문제만 효율적으로 찾아낸 방식이라기보다, 거의 모든 문제를 한 번 더 정밀하게 검토한 방식에 가깝습니다.

즉 이번 결과는 다음과 같이 구분해야 합니다.

- 정확도: 매우 높음
- 반복 재현성: 높음
- 검증 비용: 높음
- 문항 선별 효율: 아직 충분히 검증되지 않음

후속 실험에서는 113건의 딥다이브를 줄이면서도 실제로 필요했던 9건의 교정을 유지할 수 있는지가 중요합니다.

## 정답을 미리 보지 않았는가?

정답표는 풀이와 봉인 단계에서 분리했습니다.

실험은 다음 순서로 진행했습니다.

1. 실험 방법과 성공 기준을 사전등록합니다.
2. 정답표를 풀이 환경에서 제거합니다.
3. 도구 없는 조건과 도구 사용 조건을 모두 실행합니다.
4. 각 회차의 답안을 수정할 수 없도록 봉인합니다.
5. 봉인본의 SHA-256을 기록합니다.
6. 모든 회차가 끝난 뒤 정답표를 복구합니다.
7. 채점 프로그램을 한 번만 실행합니다.
8. 유리하거나 불리한 결과를 가리지 않고 전부 보고합니다.

도구 없는 조건의 평균이 38/40 이상이면 모델이 정답을 회상했을 가능성을 의심하기로 사전에 정했습니다. 실제 평균은 33/40이었으므로 이 기준에는 해당하지 않았습니다.

다만 이 검사는 회상 가능성을 완전히 배제하는 증명은 아닙니다.

## 실험 과정에서 발생한 문제

실험 과정에서 다음과 같은 운영상 문제가 있었습니다.

- SSH 연결 중단
- 모델 수용량 부족
- 일부 장시간 실행
- 실행 도중의 기계적 오류와 재실행
- F1 봉인 후 Git 커밋 지연
- 정답표 임시 격리 경로에 대한 잘못된 가정

이 문제들은 결과에서 숨기지 않고 보고서에 기록했습니다.

재실행은 점수를 확인하기 전에 발생한 기계적 실패에 한해 같은 조건으로 수행했습니다. 최종 봉인본, SEAL 파일, 스냅샷의 해시는 서로 일치합니다.

## 통계 해석상의 주의

같은 40문항을 같은 모델과 같은 도구로 세 번 반복했기 때문에 이를 서로 완전히 독립된 120개 문제로 볼 수 없습니다.

따라서 이 저장소에서는 120개의 독립 시행을 전제로 하는 단순한 p값이나 신뢰구간을 제시하지 않습니다.

세 회차 모두 만점이었다는 사실은 반복 실행의 안정성을 보여 주지만, 새로운 문제 120개에서 모두 정답을 맞혔다는 뜻은 아닙니다.

## 이 저장소를 받으면 재현할 수 있는가?

재현은 다음 세 수준으로 나눠서 보아야 합니다.

| 재현 목표 | 가능 여부 | 설명 |
|---|---|---|
| 저장된 점수·다수결·딥다이브 효과 재계산 | 가능 | 저장소와 Python만 필요 |
| 봉인본과 배포 파일 무결성 확인 | 가능 | SHA-256 목록 제공 |
| 원시 JSONL 로그 전문 확인 | 제한적 | 로그는 미포함, 해시만 제공 |
| 모델을 처음부터 다시 실행 | 조건부 가능 | 동일 모델·Codex·법률 검색 인프라 필요 |
| 파일과 출력의 비트 단위 일치 | 보장 불가 | 확률적 모델과 외부 서비스 사용 |

즉, 이 저장소만으로 기존 결과를 감사하고 재계산하는 것은 가능합니다. 하지만 모델에게 다시 문제를 풀려 완전히 같은 출력을 얻는 것은 보장할 수 없습니다.

### 1. 저장된 결과 재현

```bash
git clone https://github.com/jurisupport/korean-bar-exam-agent-eval.git
cd korean-bar-exam-agent-eval
python3 analysis/analyze_results.py
```

이 명령은 기록된 답안을 공식 정답과 다시 비교하여 점수, 다수결, 답 변경, 딥다이브 효과를 재계산합니다. 외부 패키지나 법률 DB는 필요하지 않습니다.

배포 파일이 변경되지 않았는지도 확인할 수 있습니다.

```bash
# macOS
shasum -a 256 -c manifests/SHA256SUMS.txt

# Linux
sha256sum -c manifests/SHA256SUMS.txt
```

### 2. 모델 실행 재현에 필요한 환경

처음부터 다시 문제를 풀려면 다음 환경이 필요합니다.

- Codex CLI와 `gpt-5.6-sol` 접근 권한
- `model_reasoning_effort="xhigh"`
- `korean-law` MCP
- 교재DB 8768, 실험 당시 임베딩 17,167개
- 판례DB 8770, 실험 당시 임베딩 34,788개
- 판례DB의 `EXAM_MAX_DATE=2026-01-05` 시점 차단
- 법고을·최근 판례검색 기능

교재DB와 판례DB 전체 스냅샷은 저장소에 포함되지 않았습니다. 따라서 이 인프라가 없으면 동일한 실행 조건을 구성할 수 없습니다.

### 3. 블라인드 재실행 시 주의사항

이 배포 저장소에는 채점이 끝난 정답표와 기존 답안이 포함되어 있습니다. 따라서 이 저장소 디렉터리에서 솔버를 바로 실행하면 블라인드 재현이 아닙니다.

새 실험은 다음 파일만 복사한 별도 클린룸에서 시작해야 합니다.

```text
00_fixtures/
tools/
PREREG_X.txt
RUNBOOK_X.md
```

다음 항목은 솔버가 보는 파일시스템에 두면 안 됩니다.

```text
03_answer_key/의 정답표
01_sealed/의 기존 답안
04_scoring/의 채점 결과
_runs/의 기존 스냅샷
Git 이력에 남은 정답표와 결과물
```

단순히 새 폴더로 복사하는 것만으로는 부족할 수 있습니다. 솔버가 원 저장소나 상위 디렉터리를 읽을 수 없도록, 별도 컨테이너·VM·격리된 사용자 환경을 사용하는 것이 안전합니다.

각 실행은 [실행 런북](experiment/RUNBOOK_X.md)의 순서를 따릅니다.

1. `bash tools/f_prep.sh`로 사전등록·픽스처·정답표 부재를 확인합니다.
2. Foff1·Foff2·Foff3을 각각 실행하고 봉인합니다.
3. F1·F2·F3을 서로 별도의 세션에서 실행합니다.
4. 각 회차에서 15개 1차 솔버, 다수결, 경합문항 딥다이브, 병합, 봉인을 순서대로 수행합니다.
5. 여섯 회차가 모두 봉인된 뒤에만 정답표를 복구하고 한 번 채점합니다.

회차별 딥다이브 보조 스크립트 일부는 원 실험에서 발생한 경합 목록에 맞춰져 있습니다. 새 실험에서 경합 문항이 달라지면 `f_vote.py`가 새로 생성한 목록을 기준으로 보조 스크립트를 기계적으로 일반화한 뒤, 풀이 전에 그 변경을 고정·기록해야 합니다. 현재 배포본은 결과 재계산에는 충분하지만, 임의의 새 출력을 위한 완전한 원클릭 재실행 패키지는 아닙니다.

더 자세한 환경·해시·제한사항은 [재현성과 무결성](docs/REPRODUCIBILITY.md)에 기록했습니다.

Python 표준 라이브러리만 있으면 저장된 결과를 다시 계산할 수 있습니다.

```bash
python3 analysis/analyze_results.py
```

이 스크립트는 다음을 확인합니다.

- 여섯 개 봉인본의 SHA-256
- 도구 없는 조건의 회차별 점수
- V1·V2·V3 개별 점수
- 1차 다수결 점수
- 딥다이브 전후의 답 변경
- 오답 교정과 정답 훼손 여부
- 최종 점수
- 딥다이브 적용 비율

원시 Codex JSONL 로그는 GitHub 저장소에 포함하지 않았습니다.

## 결론

이번 실험에서 도구 없는 모델은 평균 33/40을 기록했습니다. 같은 모델에 법령·판례·교재 검색, 세 가지 관점의 풀이, 다수결, 원문 중심 딥다이브를 결합하자 세 차례 모두 40/40을 기록했습니다.

그러나 최종 만점은 단순 다수결만으로 나온 것이 아닙니다. 다수결 단계의 점수는 매회 37/40이었고, 거의 모든 문항을 다시 검토하는 딥다이브가 남은 오답을 교정했습니다.

따라서 가장 정확한 결론은 다음과 같습니다.

> 충분한 계산량과 근거 원문 검증을 사용하는 전체 워크플로는 이번 공법 선택형 40문항에서 안정적인 만점을 재현했다. 다만 이를 더 적은 시간과 비용으로 달성할 수 있는지는 별도의 실험이 필요하다.
